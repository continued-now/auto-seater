## 4. Auto-Seating Algorithms

### 4.1 Simulated Annealing (SA)

**Concept:** Start with a random seating arrangement. Repeatedly make small random changes (swap two guests). Accept improvements always; accept worsening moves with a probability that decreases over time ("cooling"). This allows escaping local optima early while converging to a good solution later.

**Application to Seating:**

- **Scoring Function:** Sum of pairwise preference weights for all guest interactions. Position-weighted variants: adjacent seats (100%), directly across (60%), diagonal (30%), same table other (10%). Penalties for constraint violations (same-gender adjacency, table imbalance).

- **Neighbor Generation:**
  - Swap two random guests (simplest)
  - Move a guest to a different table
  - Swap entire table sides (saseat approach: *"it does not swap only single seats; up to an entire side will be swapped"*)
  - 90/10 split: *"spends 90% of its time optimizing within a table, only occasionally moving groups between tables"*

- **Preference Encoding:** CSV with two guest names + real-number preference weight (can be negative for conflicts). Connection matrices for relationship strength.

- **Cooling Schedule:** Geometric cooling (T = T * alpha, typically alpha = 0.95-0.999). Start temperature calibrated so ~80% of worsening moves are accepted initially.

**Implementations:**
- `saseat` (Go) -- https://github.com/shields/saseat
- `seating-planner` (Python/Flask) -- https://github.com/spookylukey/seating-planner
- `SeatingChartCreator` (Python) -- https://github.com/KatrinaE/SeatingChartCreator
- Zhijing Eu's wedding solver (Python) -- https://github.com/ZhijingEu/Wedding-Seating-Problem-Simulated-Annealing

**Pros:** Simple to implement, handles any scoring function, good at escaping local optima, near-optimal for well-tuned parameters
**Cons:** No optimality guarantee, requires tuning (cooling rate, initial temperature), non-deterministic

**Performance:** Seconds to minutes for 50-200 guests. O(1) per iteration with incremental score updates.

---

### 4.2 Constraint Satisfaction Problems (CSP)

**Concept:** Define the problem as variables (guest assignments), domains (available seats/tables), and constraints (must-sit-together, capacity limits). The solver searches for any valid assignment satisfying all constraints.

**Seating as CSP:**

```
Variables:  X_i = table assignment for guest i
Domains:    D_i = set of available tables (may be pre-restricted)
Constraints:
  - Must-sit-together: X_i = X_j (same table for couples)
  - Must-not-sit-together: X_i != X_j (different tables for conflicts)
  - Table capacity: |{i : X_i = t}| <= capacity(t)
  - Gender balance: |males_at_t - females_at_t| <= threshold
  - Accessibility: wheelchair users restricted to accessible seats
```

**Backtracking with Heuristics:**
- **Minimum Remaining Values (MRV):** Assign most constrained guest first (e.g., wheelchair users)
- **Forward Checking:** When assigning guest i to table t, eliminate t from enemies' domains
- **Arc Consistency (AC-3):** Propagate constraints aggressively to prune search space

**Solvers:** Google OR-Tools CP-SAT, MiniZinc, Choco, CPMpy

**Pros:** Natural for hard constraints, guaranteed feasible solutions, fast with good heuristics
**Cons:** Only finds feasibility (not optimality), struggles with soft preferences, exponentially slow in worst case

**Performance:** Milliseconds to seconds for 50-200 guests with moderate constraints.

---

### 4.3 Genetic Algorithms (GA)

**Concept:** Maintain a population of candidate seating arrangements. Evaluate fitness, select best for reproduction, create offspring via crossover + mutation, eliminate weak solutions. Repeat for many generations.

**Seating-Specific Details:**

- **Chromosome Encoding:**
  - Permutation-based: `[3,2,8,5,4,6,1,7]` -- guest order maps to sequential table slots
  - Group-based: `[1,1,2,3,2,1,3,2]` -- each gene = table number for that guest

- **Crossover Operators (must preserve permutation validity):**
  - Partially Mapped Crossover (PMX)
  - Order Crossover (OX1)
  - Table-preserving crossover: copy entire table assignments from one parent

- **Mutation:**
  - Swap two random guests
  - Move guest to different table
  - Reverse a segment within a table

- **Fitness Function:** Total enjoyment = sum of pairwise relationship weights. Crush=2, friend=1, acquaintance=0.5, stranger=0 (Reasonable Deviations approach).

**Implementations:**
- PerfectTablePlan (commercial) -- *"an initial population of layouts is created using heuristics... randomly mutated and spliced"*
- Megan Stiles' Seating_Chart -- https://github.com/meganstiles/Seating_Chart
- foobar98/SeatingArrangment_GA -- https://github.com/foobar98/SeatingArrangment_GA

**Pros:** Population diversity avoids local optima, any fitness function works, natural parallelism, *"usually get close to an optimal solution very quickly"* (PerfectTablePlan)
**Cons:** No optimality guarantee, crossover for permutations is tricky, many hyperparameters to tune

**Performance:** Population 16-100, converges in 100-1000 generations. Seconds to minutes for 50-200 guests.

---

### 4.4 Greedy / Hill-Climbing Local Optimization

**Concept:** Start with an initial solution. Iteratively make small changes; accept only improvements. Stop when no single change improves the solution (local optimum).

**Application to Seating:**

```
function hill_climb_seating(guests, tables, score_fn):
    current = initial_assignment(guests, tables)
    improved = True
    while improved:
        improved = False
        for each pair (i, j) at different tables:
            swap i and j
            if score(new) > score(current):
                current = new; improved = True; break
            else: undo swap
    return current
```

**Network-Flow Enhanced Variant (Oliver Wyman):**
Used for 1,500 attendees. Greedy construction (smallest groups first, network flow assignment) + bulk swaps (150 people simultaneously) using quadratic penalty scoring. Achieved near-optimal diversity *"in less than one minute."*

**Pros:** Extremely simple, fastest of all approaches, deterministic
**Cons:** Trapped by local optima, no exploration of worse solutions, poor with complex interacting constraints

**Performance:** Milliseconds for small events, under a minute for 1,500 guests with network flow.

---

### 4.5 Mixed-Integer Linear Programming (MILP)

**Concept:** Formulate seating as a mathematical program with linear objective, linear constraints, and binary decision variables. Solver guarantees provably optimal solution.

**Formulation:**

```
Decision Variables:
  x_{i,t} = 1 if guest i assigned to table t, 0 otherwise
  y_{i,j,t} = 1 if guests i AND j both at table t (linearized product)

Objective: maximize SUM(w_{i,j} * y_{i,j,t})  [total weighted interactions]

Constraints:
  - Assignment: SUM_t(x_{i,t}) = 1  for all guests
  - Capacity: SUM_i(x_{i,t}) <= capacity_t  for all tables
  - Must-together: x_{i,t} = x_{j,t}  for all t (couples)
  - Must-apart: x_{i,t} + x_{j,t} <= 1  for all t (conflicts)
  - Linearization: y <= x_i, y <= x_j, y >= x_i + x_j - 1
```

**Solvers:**
- Commercial: Gurobi ($10k+/yr), CPLEX (IBM), Xpress
- Open-source: CBC (COIN-OR), GLPK, HiGHS, SCIP, Google OR-Tools
- Python APIs: PuLP, Pyomo, gurobipy, OR-Tools

**Academic Papers:**
- Hill et al. (2024/2025) "Social Classroom Seating Assignment Problems" -- IP with valid inequalities, 320 instances up to 200 students
- Polito thesis (2021) "Seating Arrangement Optimization in COVID-19 Era" -- MILP + QUBO with Gurobi
- PuLP Wedding Example -- https://coin-or.github.io/pulp/CaseStudies/a_set_partitioning_problem.html

**Pros:** Provably optimal, hard constraints are natural, sensitivity analysis possible, mature solver technology
**Cons:** O(n^2 * T) pair variables, linearization overhead, modeling complexity, commercial solver costs

**Performance:** Seconds for 30-50 guests, minutes to hours for 100-200 guests.

---

### 4.6 Complexity Analysis

The wedding seating problem is **NP-Hard**. It is formally equivalent to the Restricted Quadratic Multiknapsack Problem (RQMKP), which is strongly NP-Hard. For n guests and T tables with capacity k, the search space is O(T^n) -- approximately 10^300 for 200 guests across 20 tables.

**Recommended Hybrid Approach for Auto-Seater:**
1. **CSP/MILP for hard constraints** -- filter infeasible assignments (accessibility, must-not-sit-together)
2. **SA or GA for optimization** -- maximize soft preferences (relationships, demographics, dietary grouping)
3. **Hill climbing for refinement** -- final local search to polish

Alternatively, **Google OR-Tools CP-SAT** bridges CSP and optimization in a unified framework.

## 5. UX Patterns

### 5.1 Drag-and-Drop

- **Dual-Panel Layout:** Guest list panel (left) + floor plan canvas (right). Most common pattern (GoJS sample, Social Tables, AllSeated). Guests dragged from list to seats.
- **Swap Mode:** Drag one seated guest onto another to swap positions. Essential for refinement.
- **Visual Feedback:** Green highlights for valid drops, red for conflicts (Table Tailor's badge system). Ghost/shadow preview during drag.
- **Canvas-Based Rendering:** HTML5 Canvas or SVG for smooth zoom/pan with 100+ elements. d3.js (Seatmap Canvas), Konva.js, or custom Canvas (istu).

### 5.2 Mobile Touch Support

- **Pinch-to-zoom** and two-finger pan on floor plan
- **Long-press** to select/move a guest (vs. tap to view details)
- **Bottom sheet** for guest list (slides up from bottom on mobile)
- **Minimum touch targets:** 44x44px (Apple HIG) / 48x48dp (Material Design)
- **Current failures:** Tables 3/16" on iPhone (Seating-Chart App), no zoom (WeddingWire)

### 5.3 Floor Plan Design

- **Table Shapes:** Round (6-12 seats), rectangular (6-20), square (4-8), cocktail/standing, head table, sweetheart table
- **Grid vs. Free-Form:** Grid snapping for aligned layouts; free-form for irregular venues. Best: toggleable snap-to-grid.
- **Fixtures & Obstacles:** Dance floor, DJ booth, stage, bar, pillars, doors, buffet stations, cake table
- **Zoom & Pan:** Essential for 100+ guest venues. Minimap for orientation. Pinch-zoom on mobile.
- **Room Dimensions:** Input real measurements (ft/m), import CAD/PDF floor plans (EventDraw), or draw walls (WeddingSeatingChart.app)

### 5.4 CSV Import

- **Field Mapping UI:** Column headers in CSV mapped to fields (name, email, dietary, group, RSVP status) via dropdown selectors
- **Validation:** Flag missing names, invalid emails, duplicate entries. Show preview before import.
- **Duplicate Detection:** Fuzzy name matching (Eventioo's "smart import"). Offer merge/skip/overwrite options.
- **Template Download:** Provide pre-formatted CSV/Excel templates (Eventioo offers Google Sheets templates)

## 6. Multi-Stage Workflow

The ideal seating arrangement process follows five stages:

### Stage 1: Guest List Management
- Import guests from CSV/Excel/Google Sheets
- Track RSVP status (invited, accepted, declined, pending)
- Record dietary restrictions, accessibility needs, plus-ones
- Group guests into households/parties
- Tag guests by social circle (family, college, work, etc.)

### Stage 2: Venue Setup
- Set room dimensions or import floor plan
- Place tables (type, shape, capacity)
- Add fixtures (dance floor, DJ, bar, stage, pillars)
- Configure table numbering scheme
- Save venue as reusable template

### Stage 3: Seating Assignment
- Auto-seed initial arrangement using algorithm (SA/GA/CSP)
- Define constraints: must-sit-together, must-not-sit-together, accessibility
- Set preferences: relationship weights, demographic balance, dietary grouping
- Manual fine-tuning via drag-and-drop
- View conflict indicators (red badges, warning icons)
- Run optimization passes on demand

### Stage 4: Review & Collaboration
- Share seating plan with partner/planner/parents via link
- Real-time collaborative editing (Google Docs-style)
- Version history with diff comparison
- Comments/annotations on specific tables or guests
- Approval workflow (planner proposes, couple approves)

### Stage 5: Export & Print
- PDF export (A4/Letter for planners, A1/poster for venue display)
- Individual table cards / place cards
- Escort card view (alphabetical guest list with table numbers)
- Shareable link for day-of digital display
- CSV export of final assignments for catering/venue staff

### Key Opportunities for Auto-Seater

1. **Auto-seating as the core differentiator.** Only 4/35 tools surveyed offer any auto-seating. None offer it in a modern web UI with real-time collaboration.

2. **Solve the cascade problem.** No tool handles last-minute changes gracefully. Re-optimization on a single change (not from scratch) would be revolutionary.

3. **Model real social dynamics.** Multi-group membership, relationship weights (not just binary), plus-one handling, divorced-parent avoidance zones.

4. **Mobile-first design.** Most tools were designed desktop-first. A responsive, touch-optimized experience would capture the mobile-majority user base.

5. **Reliable saving.** Auto-save with conflict resolution. This alone would eliminate the #1 most critical pain point.

6. **Transparent pricing.** One-time or low-cost per-event pricing. No bait-and-switch. Full-featured free tier for small events (under 50 guests).

7. **Export that works.** PDF matching screen exactly, place cards, escort cards, CSV for caterers. Tested across browsers and print sizes.
